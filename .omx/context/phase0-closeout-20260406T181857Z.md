# Task Statement

Close Phase 0 for `mcp-chrome` against four explicit goals:

0. `mcp-chrome` is stable, easy to use, and deliverable
1. all public `mcp-chrome` tool methods are live-tested
2. CoPaw re-tests the high-value tool surface and gains a better browser skill
3. the beginner install and usage manual is handoff-quality

# Desired Outcome

Reach a Phase 0 state where the browser install/restart/connect path is stable, the remaining public tool surface is validated or explicitly de-scoped, CoPaw validation has credible coverage of high-value browser operations, and the docs are strong enough for a beginner handoff.

# Known Facts / Evidence

- Browser restart persistence issue for the unpacked extension has been fixed.
- `doctor` now reports `connectivity`, `runtime.status`, and `mcp.initialize` as healthy.
- `smoke --json` currently passes end-to-end.
- CoPaw app can start and load `streamable-mcp-server`.
- CoPaw has already validated: `get_windows_and_tabs`, `chrome_navigate`, `chrome_get_web_content`, `chrome_click_element`, `chrome_fill_or_select`, `chrome_close_tabs`.
- Current validation matrix and checklists:
  - `docs/PHASE0_TOOL_VALIDATION_MATRIX.md`
  - `docs/PHASE0_COMPLETION_CHECKLIST.md`
  - `docs/COPAW.md`
  - `docs/BEGINNER_HANDOFF_zh.md`

# Constraints

- Keep changes small and independently verifiable.
- Do not regress the working browser chain.
- Prefer structured page content and JS/DOM evidence over screenshots.
- Use OMX team mode for durable parallel execution, with workers in separate worktrees.

# Unknowns / Open Questions

- Which remaining public-surface tools can be fully validated versus explicitly de-scoped for Phase 0?
- How much CoPaw interaction coverage is enough for the current Phase 0 gate?
- What is still missing from the beginner handoff package to call it deliverable?
