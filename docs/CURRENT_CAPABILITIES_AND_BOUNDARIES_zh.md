# Tabrix 当前能力面与边界

本文档专门回答一个问题：

> 当前 `Tabrix` 哪些能力是正式主线，哪些是次级能力，哪些虽然代码还在，但不应继续扩大投入？

这份文档主要给：

- AI 助手首次进入仓库时快速判断方向
- 产品经理、开发者在拆任务时控制范围
- 代码评审时识别“是否跑偏”

---

## 1. 当前正式主线

当前阶段，`Tabrix` 的正式主线能力只有这几类：

### 1.1 真实 Chrome 执行层

这是项目的根能力。

目标是：

- 复用用户真实 Chrome 会话
- 利用扩展 + native server 暴露 MCP 浏览器能力
- 在真实登录态和真实页面上下文里执行浏览器自动化

### 1.2 MCP 客户端接入

当前正式主线 transport 只有：

1. `stdio`
2. `Streamable HTTP`

所有客户端接入、文档、兼容性说明，都应围绕这两条主线展开。

### 1.3 浏览器工具与结构化读取

包括但不限于：

- 导航
- 页面读取
- 点击 / 输入
- 元素定位
- 截图
- 网络与页面辅助能力

原则是：优先强化 **Safe Tools** 和主路径稳定性，而不是扩更多边缘工具。

### 1.4 桥状态、恢复与诊断

包括：

- 浏览器是否运行
- 扩展是否连接
- bridge / command channel 是否 ready
- `status / doctor / smoke / report`
- 自动恢复与失败诊断

这是当前最关键的稳定性主线之一。

### 1.5 Record-Replay v3 内核

重点是 **内核**：

- flow / run / trigger / storage
- 稳定执行
- 可回放
- 为经验复用做基础

当前仍应继续投资 v3 内核，而不是优先做重 UI。

### 1.6 URL Experience / 经验复用方向

这还不是完全成熟的主能力，但已经是明确的主线方向：

- locatorChain
- fallback
- fingerprint
- 站点经验记忆
- 重复任务成功率提升

---

## 2. 当前应优先投入的主题

如果 AI 助手要判断“下一步做什么更值”，优先级应靠前的是：

1. `stdio` 和 `Streamable HTTP` 主链路稳定性
2. 浏览器桥状态与自动恢复
3. Popup / CLI / 文档对主线状态的统一表达
4. 核心 browser tool 的结构化失败协议
5. 真实助手链路验收
6. Record-Replay v3 内核
7. DOM 脱水、artifact、locator fallback、experience memory

---

## 3. 当前属于次级能力的部分

这些能力可以保留，但不应抢占主线优先级。

### 3.1 多客户端兼容层

兼容更多 MCP 客户端是重要的，但它的前提是主链路稳定。  
如果主链路不稳，不要为了“支持更多客户端”先扩表面兼容。

### 3.2 Remote Access 产品化细节

远程访问是强差异化方向，但前提仍然是：

- token
- 状态表达
- 桥接恢复
- 远程行为一致性

不要在底层不稳时，先堆更多外层包装。

### 3.3 文档与发布资产

这些很重要，但它们属于“信任建设层”，应服务主线，而不是替代主线。

---

## 4. 当前属于 experimental / 内部能力的部分

这些能力不是“必须删除”，但不应继续当成默认产品面。

### 4.1 扩展内智能助手

可以保留底层代码或实验入口，但不应继续作为默认产品面扩张。

### 4.2 Workflow UI / Builder 默认产品面

当前应保留 `record-replay v3` 内核，收缩未成熟 UI 暴露。

### 4.3 本地模型 / 语义索引 / 向量搜索公开能力

如果它们不直接服务当前主线，应优先评估降级、退出默认产品面或后续退场。

---

## 5. 当前不应继续扩的方向

下面这些不是当前阶段应继续投入的方向：

1. 新增第三种正式 transport
2. 把 `Tabrix` 重写成 Playwright / Stagehand 类运行时
3. 扩大扩展内 agent 产品面
4. 抢先做很重的可视化 workflow builder
5. 让 debugger / 高风险路径抢占 Safe Tools 主能力面
6. 在主链路不稳时继续扩更多边缘功能

---

## 6. 如何判断一个需求是否符合主线

看到新需求时，先问这 5 个问题：

1. 它是否增强了“真实 Chrome + MCP + 稳定执行”？
2. 它是否改善了主链路稳定性、恢复、可诊断性或复用能力？
3. 它是否会把产品表述带偏成“又一个浏览器自动化框架”？
4. 它是否会扩大当前已经判断应收缩的产品面？
5. 如果这件事不做，会不会阻塞主线发布或用户首小时成功？

如果前 2 项回答大多为“否”，而后 2 项回答偏“是”，那大概率不该现在做。

---

## 7. AI 助手进入任务前的默认判断

如果没有更高优先级明确说明，默认应该：

1. 先看是否属于主链路稳定性任务
2. 再看是否涉及桥状态 / 恢复 / Popup / CLI / 文档一致性
3. 再看是否属于 record-replay v3 / DOM / locator / 经验复用方向

不应默认把注意力放在：

- 新 transport
- 新 UI 壳子
- 与主线弱相关的历史能力面

---

## 8. 一句话边界

> 当前 `Tabrix` 应优先做深“真实 Chrome + MCP + 稳定执行 + 可恢复 + 可复用”，而不是继续把产品面做宽。
