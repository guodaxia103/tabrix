# Tabrix 公开路线图

本文档用于说明 Tabrix 的公开产品方向。

它想回答的是一个很实际的问题：

> 如果 Tabrix 继续做对事情，它接下来会长成什么样？

这是一份公开路线图，不是带日期承诺的交付清单。
它既要足够有吸引力，能让贡献者看见方向，也要足够克制，不能脱离当前代码基线和公开产品面。

## 产品方向

Tabrix 要走向的定位很明确：

> 面向 AI 助手的真实 Chrome 浏览器自动化执行层。

这意味着，我们当前并不优先把自己做成：

- 又一个通用浏览器自动化框架
- 又一个工作流 SaaS
- 又一个浏览器 IDE
- 又一个“每次都新开浏览器”的运行时

## 为什么这条路重要

当前仓库最有价值的优势，不只是“能控制浏览器”。
而是下面这组能力同时成立：

- 复用真实登录 Chrome 会话
- 通过 MCP 原生接入 AI 助手
- 本地优先，同时支持受控远程访问
- 具备日常运维所需的诊断与恢复能力

这条路线最有机会让 Tabrix 真正服务好 `Codex`、`Claude Desktop`、`Cursor`、`Cline` 等 MCP 客户端。

## Now

当前阶段的近程重点：

- 把 `Streamable HTTP` 和 `stdio` 做成最稳定的 MCP 主接入路径
- 持续提升重连、诊断、安装恢复体验
- 降低新用户首次成功和新贡献者上手门槛
- 持续保证公开文档与实际代码能力一致

用公开产品语言来说，这一阶段的核心是：
先把 Tabrix 做成一个真正“装得上、连得稳、能排障”的真实浏览器执行层。

## Next

主链路稳定后，下一批最重要的能力升级是：

- 默认输出结构化页面快照，而不是冗长噪声型页面文本
- 建立更强的真实浏览器 MCP E2E 回归与 fixture 站点覆盖
- 当 AI 助手发出命令而 Chrome 未就绪时，补齐自动拉起与恢复链路
- 通过 fingerprint、fallbackChain 和更稳定的定位策略提升动作命中率

这些能力不是锦上添花，而是决定成功率、token 成本和可预测性的关键基础。

## Later

更长期的公开方向包括：

- 面向重复网站任务的 URL Experience Memory
- 面向失败流程和 nightly 回归的 replay artifact
- 更安全的团队协作与多操作者浏览器工作流
- 建立在稳定 MCP 执行链路之上的更高层自动化抽象

这些方向都很重要，但不应跑在稳定性基础之前。

## 当前不应优先讲成什么

为了让产品叙事保持锋利，Tabrix 当前不应把自己优先讲成：

- “全能型 agent 操作系统”
- “默认工作流搭建器”
- “语义搜索产品”
- “通用浏览器 IDE”

仓库里确实存在一些相关子系统，但它们不是当前公开产品的中心叙事。

## 如何围绕路线图参与贡献

当前最值得投入的贡献方向：

- 稳定性与重连行为
- 首次成功路径与安装质量
- 结构化页面读取与提取
- fixture 站点、smoke 覆盖和回归证据
- 工具 schema 清晰度与客户端集成质量

如果你想参与某个路线图方向，建议提 issue 时至少写清：

- 用户问题是什么
- 预期行为是什么
- 会影响哪些模块
- 最小可验证成功标准是什么
