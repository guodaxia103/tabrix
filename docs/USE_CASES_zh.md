# Tabrix 典型使用场景

本文档聚焦 Tabrix 在开源冷启动阶段最真实、最值得对外表达的使用场景。

目标是：

- 让新用户快速判断 Tabrix 是否适合自己的工作流
- 强化已经成熟的公开能力面
- 避免把实验性方向包装成当前默认产品面

## 1. 研究与信息采集

**适合场景：**

- 希望 AI 助手直接读取和总结你真实浏览器里的页面
- 你已经保留了相关登录态、标签页和上下文
- 你需要截图、页面抽取或结构化读取

**典型流程：**

1. 打开或切换到目标标签页
2. 读取页面内容
3. 提取可交互元素或截图
4. 在客户端中总结结果

**Tabrix 的价值：**

- 不需要重新搭建登录环境
- 直接复用你当前真实标签页和浏览器上下文

## 2. 已登录后台自动化

**适合场景：**

- 你经常操作 CMS、工单、CRM、控制台、运营后台等需要真实登录态的系统
- 日常重复动作主要是点击、填写、提交、检查

**典型流程：**

1. 连接扩展
2. 导航到目标系统
3. 读取页面并定位目标表单或控件
4. 点击、填写、提交，并做结果确认

**Tabrix 的价值：**

- 直接复用真实登录会话
- 比“每次重开一个全新浏览器环境”更省准备成本

## 3. QA 与回归复现

**适合场景：**

- 你需要在真实浏览器会话里快速复现 UI 行为
- 你希望拿到截图、控制台输出或页面当前状态辅助分析

**典型流程：**

1. 导航到目标页面
2. 检查当前状态
3. 触发目标交互
4. 捕获截图或控制台输出

**Tabrix 的价值：**

- 能把浏览器操作和 AI 分析串起来
- `status` / `doctor` / `smoke` 与浏览器工具结合后，比较适合快速排障

## 4. 客服与运营辅助

**适合场景：**

- 你需要在支持、运营、值班过程中保留当前浏览器上下文
- 你想让 AI 辅助阅读页面、解释状态、导航到下一步动作

**典型流程：**

1. 保持你的支持/运营标签页处于打开状态
2. 让 AI 检查指定页面或状态
3. 提取内容、做摘要、跳转到下一步

**Tabrix 的价值：**

- AI 能直接利用你已经在用的浏览器上下文
- 降低人工浏览和助手协作之间的切换成本

## 5. 远程共享浏览器能力

**适合场景：**

- 你希望多个 MCP 客户端通过局域网访问同一个浏览器能力
- 你需要用 token 保护远程浏览器执行能力

**典型流程：**

1. 开启远程访问
2. 配置 token 鉴权
3. 让远程 MCP 客户端接入
4. 通过远程端点执行浏览器任务

**Tabrix 的价值：**

- 以结构化 MCP 方式暴露浏览器执行能力
- 仍然保留“真实浏览器执行层”的模型，而不是切换到另一个独立 headless 产品面

## 当前不应默认这样理解 Tabrix

今天的 Tabrix 还不适合优先被表述成：

- 一个通用工作流 SaaS
- 浏览器 IDE
- 完全产品化的 agent 操作系统
- 语义检索产品

当前公开能力边界建议结合以下文档理解：

- `PRODUCT_SURFACE_MATRIX_zh.md`
- `STABLE_QUICKSTART.md`
- `CLIENT_CONFIG_QUICKREF.md`
