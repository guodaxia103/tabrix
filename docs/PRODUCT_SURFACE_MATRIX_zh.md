# Tabrix 产品能力分层矩阵

本文档定义 Tabrix 当前公开产品面的能力边界。

它的目标是让产品规划、工程实现、文档叙事和开源协作围绕同一个问题保持一致：

> 什么属于当前公开产品，什么仍处于实验阶段，什么不应被表达成默认公开能力面？

## 为什么需要这份文档

Tabrix 仓库里的代码范围，大于当前公开产品面。

仓库中同时包含浏览器执行、MCP 传输、诊断恢复、record/replay 内核、agent 相关模块、语义/向量能力等子系统；但“代码存在”不等于“应当作为当前公开产品主承诺”。

这份矩阵是公开能力分类的 source of truth。

## 分层定义

| 层级 | 含义 | 对外表达方式 |
| --- | --- | --- |
| `GA` | 稳定、公开、可作为默认能力理解 | 可以作为默认公开产品面 |
| `Beta` | 公开可见且有战略价值，但可靠性或覆盖面仍在加强 | 可以对外提及，但要讲清边界 |
| `Experimental` | 对贡献者或未来方向有价值，但不属于默认公开承诺 | 只能谨慎提及 |
| `Internal` | 不属于公开产品面 | 不应对外作为默认能力表达 |

## 当前能力矩阵

| 能力域 | 层级 | 说明 |
| --- | --- | --- |
| 通过扩展执行真实 Chrome 浏览器能力 | `GA` | 产品根能力和核心差异化 |
| `stdio` MCP 接入 | `GA` | 当前 tier-1 主链路 |
| `Streamable HTTP` MCP 接入 | `GA` | 当前 tier-1 主链路 |
| 浏览器工具执行（读取、导航、点击、输入、截图、网络辅助） | `GA` | 日常使用的核心能力 |
| 运行时诊断（`status` / `doctor` / `smoke` / `report`） | `GA` | 建立信任和支撑运维的关键能力 |
| Token 驱动的远程访问 | `Beta` | 高价值差异化方向，但仍需继续强化 |
| Record-Replay v3 执行内核 | `Beta` | 重要方向，但还不是默认公开上手入口 |
| 可视化编辑器 | `Beta` | 有价值，但不是当前产品主叙事中心 |
| 多 MCP 客户端兼容性说明 | `Beta` | 对增长重要，但不应先于主链路稳定性扩张 |
| 经验复用 / locator fallback / 记忆式恢复方向 | `Experimental` | 很重要，但还不应作为默认公开承诺 |
| 工作流 UI 作为默认产品面 | `Experimental` | 代码存在，但不应定义当前公开产品叙事 |
| Agent / sidepanel 助手产品面 | `Experimental` | 仓库中存在，但不是当前默认公开主能力 |
| 本地语义索引 / 向量重能力 | `Experimental` | 技术探索价值大，但不是当前公开主定位 |
| 内部评审、验收证据、夜测、PM 执行系统 | `Internal` | 治理材料，不属于公开产品面 |

## 当前应持续重复的公开产品叙事

今天的 Tabrix，最应该稳定重复的表达是：

1. 把用户真实 Chrome 会话变成 AI 可调用执行层
2. 通过 MCP 暴露浏览器自动化能力
3. 本地优先，同时支持远程调用、状态诊断与恢复

今天不应优先对外讲成：

- 一个通用工作流 SaaS
- 浏览器 IDE
- 全能型 agent 操作系统
- 语义检索产品

## 规划规则

做后续产品与工程规划时，默认遵守：

1. 先提升 `GA` 稳定性，再扩张 `Experimental`
2. 不把 `Experimental` 子系统当成默认公开产品面宣传
3. 对外可见但仍不够稳的能力，应归为 `Beta`，不要提前写成 `GA`
4. 主要服务未来方向的子系统，在没有稳定用户路径前，不应进入 headline 产品叙事
