## 1. 文档目标

本文档是 `Tabrix` 的总产品规格书候选稿。

它用于统一回答以下问题：

1. `Tabrix` 到底是什么产品。
2. `Tabrix` 当前优先服务谁，不优先服务谁。
3. `Tabrix` 当前公开承诺的能力边界是什么。
4. 什么条件下可以称为“可交付”，什么条件下可以称为 `GA`。
5. `T0~T15` 这些任务和产品目标之间是什么关系。

本文档的职责是成为产品层 source of truth。

这意味着：

- `README` 负责对外说明。
- `ROADMAP` 负责公开方向。
- `RELEASE_READINESS_*` 负责发布门槛。
- `T*` 任务规格负责实现路线。
- **本 PRD 负责定义产品本身。**

---

## 2. 一句话定义

> `Tabrix` 是一个把用户真实 Chrome 会话变成 AI 可调用执行层的浏览器控制产品。

这句话有四个约束：

1. 基于 **真实 Chrome**，不是默认重开隔离浏览器。
2. 面向 **AI 助手调用**，不是只做脚本框架。
3. 重点是 **执行层 + 理解层**，不是单纯网页转文本。
4. 默认强调 **本地优先、低痕迹、可恢复、可治理**。

---

## 3. 产品问题定义

## 3.1 要解决的问题

当前 AI 助手在浏览器里普遍存在四个问题：

1. 无法直接复用用户已经登录的真实浏览器会话。
2. 浏览器读取结果噪音太大，token 成本高，理解质量不稳定。
3. 真实后台、复杂页面、长流程任务的动作命中率和恢复能力不足。
4. 即使能控制浏览器，也缺少诊断、恢复、回归、审计这些产品级能力。

`Tabrix` 的目标是把这四个问题变成产品化能力，而不是零散脚本能力。

## 3.2 不解决的问题

`Tabrix` 当前不优先解决以下问题：

1. 做成通用无头爬虫框架。
2. 做成浏览器 IDE。
3. 做成默认工作流 SaaS。
4. 做成以语义检索或知识库为中心的产品。
5. 在没有稳定执行和安全边界前，先扩展成全行业全页面覆盖平台。

## 4. 目标用户

## 4.1 当前主用户

当前主用户应定义为：

1. 需要让 `Codex`、`Claude Desktop`、`Cursor` 等 MCP 客户端直接操作真实 Chrome 的高级用户。
2. 需要在已登录后台、研究页面、运营页面中完成实际任务的开发者或自动化用户。
3. 需要真实会话复用，而不是“再开一个浏览器”的 AI 助手用户。

## 4.2 次级用户

次级用户包括：

1. 需要团队内受控远程访问的高级用户。
2. 需要把浏览器能力纳入内部工作流的技术团队。
3. 对 replay、evidence、memory、policy 有进一步要求的企业用户。

## 4.3 当前不应过度承诺的用户

当前不应把以下人群当成已完全覆盖的默认用户：

1. 大规模企业自动化平台采购方。
2. 高合规行业默认落地用户。
3. 需要跨浏览器、跨 OS、跨团队稳定 SLA 的大组织。

这些方向有价值，但不应在当前阶段被表述成默认已满足。

## 5. 核心场景

## 5.1 首次成功场景

用户在自己的机器上：

1. 安装 CLI。
2. 加载扩展。
3. 建立浏览器连接。
4. 在 MCP 客户端中调用浏览器能力。
5. 看到 AI 助手能在真实浏览器里读取页面、点击、输入、截图、继续执行。

这是最基础的产品场景。

## 5.2 日常执行场景

用户在已登录浏览器上下文中：

1. 读取当前页面。
2. 理解页面任务模式、页面角色、主区域。
3. 聚焦高价值对象。
4. 执行动作。
5. 必要时读取更多细节而不是整页重读。

这对应当前 `T5` 主线。

## 5.3 远程接入场景

用户在本地浏览器上保留会话，由另一个 MCP 客户端通过受控方式调用浏览器能力。

该场景当前是高价值方向，但安全、审计、策略能力尚未完整。

## 5.4 故障恢复场景

连接、桥接、扩展、运行时出现问题时，用户能通过：

- `status`
- `doctor --fix`
- `smoke`
- `report`

快速判断问题位置并恢复。

这是产品可信度的重要组成部分。

## 6. 产品能力边界

当前公开边界以 [PRODUCT_SURFACE_MATRIX_zh.md](./PRODUCT_SURFACE_MATRIX_zh.md) 为准，但产品层应理解为：

## 6.1 当前 GA 能力

1. 真实 Chrome 执行能力。
2. `stdio` / `Streamable HTTP` MCP 主接入路径。
3. 基础浏览器工具执行能力。
4. 运行时诊断与恢复闭环。

## 6.2 当前 Beta 能力

1. Token 驱动远程访问。
2. Record-Replay v3 相关能力。
3. 更高层的页面理解和复杂工作流能力。

## 6.3 当前 Experimental 能力

1. 经验复用与记忆化恢复。
2. 更高层 agent 产品面。
3. 大规模语义 / 向量能力。
4. 全行业复杂页面家族适配。

---

## 7. 产品原则

## 7.1 真实会话优先

默认优先复用真实浏览器，而不是重新建立自动化环境。

## 7.2 低痕迹接管

默认避免高扰动、高识别的接管方式成为主叙事。

## 7.3 理解先于动作

复杂页面上，AI 不应直接面对整页噪音，而应通过分层理解后再执行动作。

## 7.4 恢复能力属于产品本体

`status / doctor / smoke / report` 不是附属工具，而是产品核心能力的一部分。

## 7.5 安全与治理不是后补附件

远程访问、安全策略、敏感站点保护、审计能力，不应长期停留在“以后再说”。

---

## 8. 理解层产品要求

`T5` 代表 `Tabrix` 的任务导向网页理解主线。

其目标不是“网页转 JSON”，而是让 AI 助手在复杂页面上：

1. 更快找到当前任务相关区域。
2. 更准识别当前页面是什么。
3. 更省 token 地决定下一步动作。

理解层至少包括：

1. `taskMode`
2. `pageRole`
3. `primaryRegion`
4. `highValueObjects`
5. `L0 / L1 / L2`

产品上，这些能力必须服务“更快、更准、更省 token”的浏览器操作，而不是成为新的冗余协议输出。

## 9. 北极星指标

当前 PRD 建议至少跟踪以下指标：

1. 首次成功率
2. 真实任务成功率
3. 页面理解正确率
4. token 成本中位数 / P95
5. 回归稳定性
6. 运行时恢复成功率

在这些指标未建立之前，不能只用“功能存在”替代产品成熟度判断。

## 9.1 指标口径

为避免“看起来有指标、实际上无法比较”，`v1` 先给出最小计算口径：

### 首次成功率

定义：

- 新用户在首次安装后的 `30` 分钟内，
- 至少完成一次：
  - 建立连接
  - MCP 客户端成功调用
  - 浏览器真实执行成功

建议口径：

- 分子：满足上述闭环的会话数
- 分母：进入首次安装流程的会话数

### 真实任务成功率

定义：

- 用户或回归任务在真实浏览器中完成目标动作链，
- 且结果满足预期，不以“工具调用返回成功”替代任务成功。

建议口径：

- 分子：端到端任务完成次数
- 分母：真实任务尝试次数

### 页面理解正确率

定义：

- 对指定 fixture / baseline 页面，
- `taskMode / pageRole / primaryRegion / highValueObjects` 达到预期。

建议口径：

- 至少拆成四项独立统计
- 不能只给一个笼统“理解通过率”

### token 成本

定义：

- 以 `read_page` 主链输出为统计对象，
- 跟踪 `p50 / p95` token 成本

建议口径：

- 固定 fixture 集
- 固定模型 / tokenizer 版本
- 回归阈值按 release 分支对比

### 回归稳定性

定义：

- 在固定公开基线与 E2E 集合上，
- 最近窗口内通过率和 flaky 情况

建议口径：

- 近 `30` 次 CI
- 近 `7` 天 nightly

### 运行时恢复成功率

定义：

- bridge / extension / runtime 故障注入后，
- 是否能恢复到可继续执行状态

建议口径：

- `doctor --fix`
- 自动拉起
- 原请求续跑

至少拆成独立恢复类型统计。

## 10. 发布级门槛

发布是否达到“可交付”不由单个功能决定，而由以下四类条件共同决定：

1. 架构中立性
2. 真实验收面
3. 可靠执行与可观测性
4. 企业级非功能要求

## 11. 版本判断

## 11.1 `v2.x`

`v2.x` 可以继续作为“产品能力逐步成型”的阶段版本，但不应轻易把任何单点功能包装成整体 GA。

## 11.2 `v3.0`

只有在以下条件都成立时，才应考虑 `GA` 级别表述：

1. 核心理解和执行主链稳定。
2. 真实 E2E 与回归体系成立。
3. 安全、策略、审计、敏感域保护具备明确能力。
4. 至少不止一个页面家族被产品级验证。
5. 发布门槛与北极星指标有持续数据支撑。

## 11.3 版本策略

当前建议采用以下版本语义：

### `v2.0.x`

定位：

- 主线能力收口期
- 重点是：
  - 安装、连接、恢复
  - 基础执行层
  - 理解主链起步

不应表述为：

- 全行业通用 GA
- 企业级完整交付

### `v2.1.x`

定位：

- 架构收口 + 理解层完结 + 执行可靠性增强

应重点完成：

- `T5` 完结
- `T7/T8/T9/T10` 中至少一部分进入可用状态
- 发布门槛和北极星指标开始稳定化

### `v2.2.x` 及之后

定位：

- 企业化和持续质量增强期

重点包括：

- 多页面家族扩展
- 安全 / 合规 / 审计增强
- 团队级共享能力
- 夜测与持续验收

### `v3.0`

只有在产品叙事、能力覆盖、非功能要求、发布门槛都达到一致时，才应作为 GA 候选。

## 11.4 兼容性承诺

在 PRD 层，当前至少应明确以下兼容性原则：

1. `MCP tool schema` 不能在无说明的情况下随意破坏。
2. `MAJOR` 版本必须显式列出破坏性变更。
3. 扩展与 native-server 版本错配时，行为必须有文档说明。
4. 新理解字段可以增量扩展，但不能静默破坏已稳定契约。

## 12. 任务体系与产品体系的关系

`T0~T15` 是实现路线，不是产品定义本身。

必须明确：

1. 任务号可以帮助管理实现顺序。
2. 任务号不能替代产品定义。
3. 同一个 `T` 编号不应漂移成多个语义。
4. 产品对外叙事不应直接等于内部任务结构。

产品 SoT 应优先是本 PRD，其次才是路线图和任务规格。

## 12.2 使用规则

在产品层，任务映射表必须遵守：

1. 一个任务号只对应一个主语义。
2. 子任务可以细分，但不能反向改写主任务含义。
3. 任何任务号语义变更，都必须先更新治理文档和主表 SoT。
4. 对外文档优先使用“产品能力域”，内部实现材料再引用 `T*` 编号。

## 13. 当前阶段判断

截至当前，`Tabrix` 更准确的产品状态应表达为：

1. 在“真实 Chrome + MCP 执行层”方向上已具备清晰产品形态。
2. 在 GitHub family 与部分理解主链上，已经形成产品级子能力。
3. 整体仍未达到“全行业通用、企业级 GA”的状态。

因此，当前最重要的工作不是盲目扩张能力面，而是：

1. 收口架构边界
2. 做强真实验收面
3. 把理解层、执行层、安全层、恢复层逐步补齐
