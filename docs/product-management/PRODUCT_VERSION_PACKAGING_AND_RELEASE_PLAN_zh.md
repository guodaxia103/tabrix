## 1. 文档目标

本文档用于回答以下问题：

1. `T1-T16` 这些任务分别更适合进入哪个版本
2. 某个版本的发版门槛是什么
3. 哪些任务是该版本的 `Blocker`
4. 哪些任务可以进入下一版，而不阻塞当前版本发布

本文件不替代主任务系统文档，而是作为“任务到版本”的装箱规则文档。

---

## 2. 使用原则

版本装箱遵守以下原则：

1. 先按能力任务定义问题，再按版本装箱
2. 版本号不能替代任务本身
3. 同一任务如果跨多个版本演进，首版先交最小可发布能力
4. 未完成 `Blocker` 不进入目标版本“可发版”状态
5. 版本边界以用户可感知价值、风险收口和验收成熟度为准

## 3. 当前版本策略

### 3.1 `v2.0.9`

定位：

- 稳定性强化版
- 真实链路与真实验收收口版
- 开源协作与发布治理收口版

适合进入的任务类型：

- transport 稳定
- 恢复链路
- 统一错误语义
- 真实验收门禁
- 客户端接入
- 第三方复用边界
- E2E 回归
- nightly 报告

### 3.2 `v2.1.0`

定位：

- 结构化输出与经验复用增强版
- 真正拉开 Tabrix 差异化能力的版本

适合进入的任务类型：

- DOM 脱水
- DOM artifact
- locator/fallbackChain/fingerprint
- URL Experience Memory
- replay artifact

### 3.3 `v2.0.x`

定位：

- 非主链功能收口与产品面整理

适合进入的任务类型：

- 默认产品位收缩
- 实验能力降级
- 工作流 UI 降级
- 本地模型/语义索引退场评估

## 5. 版本发布门槛

## 5.1 `v2.0.9` 发布门槛

至少满足：

1. `T1-T7` 达到 `done` 或具备不阻塞发版的明确范围收口
2. `fast / full` 真实验收边界明确
3. 至少两条主链路可验证：
   - `Streamable HTTP`
   - `stdio`
4. 至少一个真实助手链路可稳定通过：
   - `Codex -> Claude CLI -> Tabrix MCP 服务 -> 真实 Chrome`
5. 客户端接入与错误语义文档已同步
6. 第三方复用边界与 NOTICE 流程已建立

## 5.2 `v2.1.0` 发布门槛

至少满足：

1. `v2.0.9` 主链路已稳定
2. `T8-T12` 中至少 3 项形成真实可发布能力
3. DOM 脱水与 artifact 不破坏现有主线工具调用
4. locator/fallbackChain 有实际命中率提升证据
5. 至少一类站点任务在经验复用下看到 token 成本或成功率收益

## 6. 任务跨版本规则

当任务可能跨版本演进时，按以下原则处理：

1. 首版先交最小可发布能力
2. 后续增强继续沿用原任务编号，但回写到新版本装箱中
3. 不允许把“完全没做完”的任务包装成“已发布能力”

示例：

- `T8`
  - `v2.1.0`：先交 `compact / normal / full` 的最小可用版
  - 后续版本：再补更强 DOM abstraction

- `T11`
  - `v2.1.0`：先交推荐与复用
  - 后续版本：再补自动学习与更复杂策略

---

## 7. 版本变更时必须更新什么

每次版本装箱变动，至少同步更新：

1. 本文档版本号
2. 主任务系统文档中的任务状态或建议版本
3. 飞书文档
4. 如已接近发版，还要同步：
   - release note 候选项
   - release readiness checklist
   - acceptance matrix

## 8. 当前建议的最近版本路线

### 8.1 近期版本顺序

1. `v2.0.9`
2. `v2.1.0`
3. `v2.0.x` 中的产品面收缩与退场动作视情况穿插

### 8.2 当前推荐解释

- `v2.0.9` 先解决“能不能稳定接、稳定跑、稳定验收”
- `v2.1.0` 再解决“能不能更省 token、更稳定复用、更容易复盘”
- 产品面收缩类任务可以穿插，但不能打断主链路
