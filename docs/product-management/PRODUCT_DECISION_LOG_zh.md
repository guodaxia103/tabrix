## 1. 文档目标

本文档用于记录 `Tabrix` 当前阶段已经明确达成的产品级决策，避免：

1. 关键边界只存在于聊天记录中
2. 同一个问题被反复重新讨论
3. `Codex` 执行任务时缺少稳定的上位约束
4. 发布、裁剪、复用、验收等关键判断在后续被无意推翻

本日志只记录已经明确达成共识的决策，不记录开放讨论中的想法。

## 3. 当前有效决策

## PD-2026-001：Tabrix 的核心定位不是“又一个浏览器自动化工具”

- 决策内容：
  - `Tabrix = 接管用户真实 Chrome 的 MCP 原生执行层，并以低痕迹方式完成稳定、可复用、可回放的浏览器自动化。`
- 决策原因：
  - 这一定义同时来自 README、产品定位文档、项目评审和代码现状。
- 影响范围：
  - 产品叙事
  - 任务优先级
  - 功能裁剪
  - 对外文档表达
- 后续动作：
  - 所有新增功能都要先判断是否增强“真实 Chrome / MCP / 低痕迹 / 稳定复用回放”这条主线。
- 是否允许未来调整：
  - 允许，但只能在大版本产品转向时调整。

## PD-2026-002：当前只承认两条正式主线 transport

- 决策内容：
  - 当前阶段只把 `stdio` 和 `Streamable HTTP` 视为正式主线 transport。
- 决策原因：
  - 项目文档已经明确，继续扩展更多连接形态会分散排障、测试和产品表达精力。
- 影响范围：
  - transport 文档
  - 接入文档
  - 测试与验收
  - 任务优先级
- 后续动作：
  - `T1`、`T4`、`T7`、`T13` 必须围绕这两条链路收口。
- 是否允许未来调整：
  - 允许，但前提是这两条链路先稳定。

## PD-2026-003：远程 Streamable HTTP 优先级高于本机 stdio 之外的其它模式

- 决策内容：
  - 连接方式优先级明确为：
    1. 远程 `Streamable HTTP`
    2. 本机 `stdio`
- 决策原因：
  - 远程控制与 LAN 工作流是 Tabrix 的重要差异化能力。
- 影响范围：
  - 产品路线
  - 验收重点
  - 客户端接入策略
- 后续动作：
  - 优先为远程 HTTP 补齐回归、恢复、文档、nightly 和真实验收。
- 是否允许未来调整：
  - 短期内不建议调整。

## PD-2026-004：浏览器恢复必须由 MCP 服务状态机主导

- 决策内容：
  - 浏览器桥接状态必须由 MCP 服务统一维护，不能依赖助手脚本临时判断或掩盖状态问题。
- 决策原因：
  - 状态分散和误判是当前恢复链路不稳定的根因之一。
- 影响范围：
  - `T2`
  - `status / doctor / report`
  - 恢复相关返回结构
- 后续动作：
  - 收口 daemon、browser、native host、bridge、command channel 的状态语义。
- 是否允许未来调整：
  - 不建议调整。

## PD-2026-005：只有在收到浏览器自动化任务时才允许按需拉起浏览器

- 决策内容：
  - 不允许在没有浏览器自动化请求的前提下偷偷启动浏览器。
- 决策原因：
  - 这会影响用户感知、干扰真实浏览器使用，也违背低打扰原则。
- 影响范围：
  - `T2`
  - 自动恢复设计
  - CLI / doctor 行为
- 后续动作：
  - 恢复链路必须围绕“任务触发 -> 状态判断 -> 按需恢复”来实现。
- 是否允许未来调整：
  - 原则上不建议。

## PD-2026-006：真实助手链路验证是正式验收，不是临时测试说明

- 决策内容：
  - 涉及浏览器主链路的任务，默认要求至少一条真实验收路径：
    - `Codex -> Claude CLI -> Tabrix MCP 服务 -> 真实 Chrome`
- 决策原因：
  - 仅靠单元测试、模块测试或本地 smoke 无法证明真实产品链路成立。
- 影响范围：
  - `T1`
  - `T2`
  - `T3`
  - `T4`
  - `T5`
  - `T7`
  - `T13`
- 后续动作：

- 是否允许未来调整：
  - 可扩展，但不应删除。

## PD-2026-007：Safe Tools 是默认主能力面，Debugger Tools 不得伪装成普通能力

- 决策原因：

- 影响范围：
  - `T3`

- 后续动作：
  - 统一工具保护协议与风险表达。
- 是否允许未来调整：
  - 仅允许细化，不建议推翻。

## PD-2026-015：Popup 顶层只保留 stdio 与远程（Streamable HTTP）两种正式连接方式

- 决策内容：
  - Popup 顶层不再把 `本机 HTTP` 作为第三种产品模式对外暴露。
  - 顶层只保留 `stdio` 与 `远程（Streamable HTTP）`。
  - `本机 localhost HTTP` 若继续存在，只能作为 `远程（Streamable HTTP）` 下的实现细节或说明。
  - `远程（Streamable HTTP）` 默认选中，并默认处于安全开启、带 Token 的可复制状态。
- 决策原因：
  - 继续暴露 `本机 / stdio / 远程` 三模式，会和已确定的两条正式主线 transport 形成产品口径冲突。
- 影响范围：
  - `T1`
  - Popup UI
  - 配置复制文案
  - 远程访问默认策略
- 后续动作：
  - 在 `T1` 中收口连接方式、默认远程开启和 Token 就绪体验。
- 是否允许未来调整：
  - 短期内不建议调整。

## PD-2026-016：客户端主列表展示“有效活跃客户端”，不是原始会话快照

- 决策内容：
  - Popup 中的客户端列表不再等同于后端原始 `Streamable HTTP` session dump。
  - 主列表默认只展示 `active` 客户端或活跃 MCP 会话。
  - 需要补最小状态语义和治理策略，至少覆盖 `active / stale / disconnected`。
  - stale 会话必须自动清理或支持一键清理；手动断开的条目不能继续停留在主列表。
- 决策原因：
  - 当前重复重连会把同一客户端堆积成几十条历史 session，用户无法判断哪些是真正有效的客户端。
- 影响范围：
  - `T1`
  - `/status` 语义
  - Popup 客户端列表
  - 会话治理与排障体验
- 后续动作：
  - 在 `T1` 中重定义客户端列表数据语义，并补最小状态与清理策略。
- 是否允许未来调整：
  - 允许，但前提是主列表仍然必须表达“当前有效活跃客户端”。

## PD-2026-017：localhost HTTP 继续保留，但只能作为 Streamable HTTP 的本机实现细节

- 决策内容：
  - `127.0.0.1 / ::1` 的本机 HTTP 请求继续保留免 Token。
  - 非本机 HTTP 请求必须携带 `Authorization: Bearer <token>`。
  - `localhost HTTP` 不能作为第三种独立产品模式对外暴露，只能作为 `远程（Streamable HTTP）` 下的本机实现细节存在。
  - UI、状态接口与文档必须显式区分：
    - `本机 HTTP（免 Token）`
    - `远程 HTTP（需 Token）`
    - `stdio`
  - `127.0.0.1` 的本机 HTTP 客户端不得被误表达为“远端客户端”。
- 决策原因：
  - 当前实现已经保留 localhost 豁免 Bearer 的行为，直接改变默认策略回归风险较高。
  - 真正的问题不在于 localhost 是否免 Token，而在于产品语义不清，容易把“本机 HTTP”误看成“远程 HTTP”。
- 影响范围：
  - `T1`
  - Popup UI
  - `/status` 与客户端列表语义
  - transport / client config / troubleshooting 文档
- 后续动作：
  - 在 `T1` 中明确补齐 localhost 与远端 HTTP 的鉴权和展示规则。
  - 后续如需提升安全级别，可单独评估“localhost 也要求 token”的严格模式，但不作为当前默认策略。
- 是否允许未来调整：
  - 允许，但必须先评估本机调试、CLI 接入和回归风险。
