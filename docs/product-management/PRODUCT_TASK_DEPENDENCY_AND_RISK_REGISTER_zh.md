## 1. 文档目标

本文档用于补齐 `Tabrix` 当前任务系统中最容易缺失的一层：

1. 每个任务依赖什么前提
2. 每个任务最大的风险是什么
3. 哪些任务会阻塞其它任务
4. 哪些任务虽然重要，但不应提前启动

本文件与主任务系统文档、版本装箱文档配套使用，不替代完整任务定义。

## 2. 使用规则

### 2.1 这份文档解决什么问题

后续把任务投喂给 `Codex` 时，如果只给“目标、范围、验收标准”，通常会有两个问题：

1. 在错误前提下直接开工
2. 同时做了一个被上游阻塞的任务，浪费时间

本清单的作用就是在执行前，先把任务依赖和风险看清楚。

### 2.2 风险等级

统一使用以下等级：

- `R1`：低风险，可正常推进
- `R2`：中风险，需要关注依赖或边界
- `R3`：高风险，容易导致返工或主链路不稳定

### 2.3 依赖状态

统一使用以下依赖状态：

- `hard_dependency`：必须先满足，否则不建议启动
- `soft_dependency`：有最好，没有也能先做一部分
- `parallelizable`：可以并行，不直接阻塞

## 4. 任务依赖与风险清单

## T1 主线连接方式与客户端会话模型收口

- 依赖：
  - `T6`：`soft_dependency`
  - 当前 transport 文档、Popup 文案与 CLI 行为一致性：`hard_dependency`
  - SessionRegistry 与 `/status` 当前客户端语义梳理：`hard_dependency`
- 风险等级：`R3`
- 主要风险：
  - 远程默认开启如果处理不好，会引入鉴权暴露或错误安全感
  - `本机` 模式下线后，如果没有把 localhost HTTP 收回到远程语义，会造成新的文档和 UI 歧义
  - 直接把原始 session 继续当客户端展示，会导致“67 个客户端”这类误导持续存在
  - stale/disconnected 清理策略不清，会出现主列表错误删活跃会话或继续堆积无效会话
- 启动条件：
  - transport、Popup、status、CLI、error codes 相关文档已核对
- 不建议提前做的原因：
  - 如果连“产品模式”和“原始会话”的边界都没看清，容易把 UI 改对了，但底层语义仍然是错的

## T2 助手命令恢复链路

- 依赖：
  - `T1`：`hard_dependency`
  - 浏览器桥接状态语义：`hard_dependency`
- 风险等级：`R3`
- 主要风险：
  - 状态源不统一，恢复逻辑越做越乱
  - 不小心变成“偷偷启动浏览器”
  - 恢复成功与失败返回不一致，AI 客户端很难稳定处理
- 启动条件：
  - 双链路错误语义和状态语义已基本收口
- 不建议提前做的原因：
  - 如果 `T1` 未收口，`T2` 会在错误地基上堆逻辑

## T3 核心 browser tool 统一保护协议

- 依赖：
  - `T1`：`hard_dependency`
  - 工具分层文档与错误码文档：`hard_dependency`
- 风险等级：`R3`
- 主要风险：
  - 不同工具各改一套，反而更不一致
  - 把 Debugger Tool 误包装成 Safe Tool
  - 改了错误结构但文档和客户端没同步
- 启动条件：
  - 关键工具清单、分类边界已确认
- 不建议提前做的原因：
  - 没有统一错误语义前，容易做成零散修补

## T4 真实验收门禁化

- 依赖：
  - `T1`：`hard_dependency`
  - `T2`：`soft_dependency`
  - `T3`：`hard_dependency`
  - 已有 acceptance 资产：`hard_dependency`
- 风险等级：`R3`
- 主要风险：
  - 门禁定义混乱，团队无法判断“什么叫可发版”
  - 把人工协作型能力混入无人值守通过率
  - 第二客户端引入后，出现“一个通过一个不通过”的复杂结论
- 启动条件：
  - 至少一条主链路和一批主线工具已基本稳定
- 不建议提前做的原因：
  - 如果基础行为还在变，门禁会频繁重写

## T5 Codex 客户端一等接入

- 依赖：
  - `T1`：`hard_dependency`
  - `T4`：`soft_dependency`
  - 官方 skill 路由策略：`hard_dependency`
- 风险等级：`R2`
- 主要风险：
  - 文档看起来清楚，但真实接入仍有隐藏前提
  - Codex 路由不稳定，先走别的 browser skill
  - 为 Codex 特化过度，影响其他客户端
- 启动条件：
  - Codex 当前接入链路可实际验证
- 不建议提前做的原因：
  - 如果基础 transport 和错误语义未稳定，文档会很快过期

## T6 第三方复用矩阵与 NOTICE 流程

- 依赖：
  - 当前开源复用边界文档：`hard_dependency`
- 风险等级：`R1`
- 主要风险：
  - 上游许可证理解不准确
  - 把“设计参考”和“代码来源”混淆
- 启动条件：
  - 可访问重点开源项目信息和当前文档
- 不建议提前做的原因：
  - 无，适合最早做

## T7 真实 MCP E2E fixture 站点与回归框架

- 依赖：
  - `T1`：`hard_dependency`
  - `T3`：`soft_dependency`
  - 现有 acceptance / smoke 资产：`hard_dependency`
- 风险等级：`R3`
- 主要风险：
  - fixture 做得太玩具化，不能代表真实问题
  - 入口不是真实 MCP，而是内部模块测试伪装
  - case 太多但不可维护
- 启动条件：
  - 主链路和主线工具至少能稳定跑一批 case
- 不建议提前做的原因：
  - 如果主链路还在变，E2E 框架会频繁返工

## T8 DOM 脱水与极简 JSON 树输出

- 依赖：
  - `T1`：`soft_dependency`
  - `T3`：`soft_dependency`
  - `T9`：`soft_dependency`
- 风险等级：`R3`
- 主要风险：
  - 输出变更影响现有客户端
  - 过度压缩导致关键信息丢失
  - 结构变漂亮了，但对 click/fill 没实际帮助
- 启动条件：
  - read_page 当前消费方已梳理清楚
- 不建议提前做的原因：
  - 如果主线工具和 artifact 路线还没稳定，结构容易推倒重来

## T9 DOM artifact 接入 execution session

- 依赖：
  - `T8`：`soft_dependency`
  - session / result normalizer 现有链路：`hard_dependency`
- 风险等级：`R2`
- 主要风险：
  - artifact 生命周期失控
  - 清理策略不明确，导致残留或引用失效
- 启动条件：
  - execution session 与 artifactRefs 现状已梳理清楚
- 不建议提前做的原因：
  - 如果 DOM 输出方案未确定，artifact 结构也会跟着变

## T10 locator 排名 / fallbackChain / fingerprint

- 依赖：
  - `T3`：`hard_dependency`
  - `T8`：`soft_dependency`
  - 现有 selector engine：`hard_dependency`
- 风险等级：`R3`
- 主要风险：
  - 候选链太重，影响性能
  - 命中看似变多，误点也变多
  - 不可解释的回退会让调试成本上升
- 启动条件：
  - selector engine 行为与当前工具消费方已梳理
- 不建议提前做的原因：
  - 如果工具失败语义未收口，回退逻辑很难定义清楚

## T11 URL Experience Memory v1

- 依赖：
  - `T10`：`soft_dependency`
  - `record-replay v3`：`hard_dependency`
- 风险等级：`R3`
- 主要风险：
  - 经验命中错误导致重复失败
  - 数据结构设计过重，首版无法落地
  - 经验库污染主流程
- 启动条件：
  - v3 内核与 locator 基础能力已能稳定提供输入
- 不建议提前做的原因：
  - 上游定位与结构化输出不稳时，经验库没有稳定素材

## T12 失败流程 replay artifact

- 依赖：
  - `T9`：`soft_dependency`
  - `T13`：`soft_dependency`
  - rrweb 引入边界：`hard_dependency`
- 风险等级：`R2`
- 主要风险：
  - replay 成本过高
  - 敏感信息处理不清晰
  - 证据很多但不可用
- 启动条件：
  - 失败证据链的目标和 artifact 体系基本明确
- 不建议提前做的原因：
  - 如果报告链路和 artifact 结构都没定，回放价值会被削弱

## T13 nightly 稳定性与回归报告自动化

- 依赖：
  - `T4`：`soft_dependency`
  - `T7`：`soft_dependency`
- 风险等级：`R2`
- 主要风险：
  - nightly 噪音过大
  - 报告输出不稳定，难以持续比较
  - 环境波动导致假失败
- 启动条件：
  - 至少一批核心回归 case 已稳定
- 不建议提前做的原因：
  - 如果没有稳定可复用的验收清单，nightly 很快失真

## T14 智能助手退出默认产品面

- 依赖：
  - 功能裁剪结论：`hard_dependency`
- 风险等级：`R1`
- 主要风险：
  - UI 入口收缩后出现死链接
  - 内部团队仍依赖某些隐藏入口
- 启动条件：
  - 相关入口和依赖链已盘点
- 不建议提前做的原因：
  - 可随主链路并行，但应避免打断主线研发

## T15 工作流 UI 降级，保留 v3 内核

- 依赖：
  - 功能裁剪结论：`hard_dependency`
  - record-replay v3 内核确认保留：`hard_dependency`
- 风险等级：`R2`
- 主要风险：
  - UI 降级时误伤 v3 内核能力
  - 旧入口残留导致用户误解
- 启动条件：
  - workflow 入口与内核耦合点已盘点
- 不建议提前做的原因：
  - 可与主链路并行，但不应占用主线关键资源

## T16 本地模型 / 语义索引 / 向量搜索退场评估

- 依赖：
  - 功能裁剪结论：`hard_dependency`
  - 相关模块依赖盘点：`hard_dependency`
- 风险等级：`R2`
- 主要风险：
  - 某些模块被主链路间接依赖，贸然删会误伤
  - 入口收缩与实际代码状态不一致
- 启动条件：
  - 相关代码、依赖、入口、测试已盘点
- 不建议提前做的原因：
  - 可并行做评估，但不应在未完成盘点前直接删改
