# Tabrix AI 助手开发通用规则

本文档是 Tabrix 项目的公开协作规则入口。

目标不是限制具体实现方式，而是让任何进入公开仓库的 AI 助手或贡献者，都能以一致、稳定、低回归的方式工作。

本仓库默认要求所有代码开发遵循 `karpathy-guidelines`：

- 先想清楚再动手
- 先做最小必要改动
- 明确成功标准并用验证闭环收口
- 能修根因时，不接受只屏蔽症状的临时止血

这份规则适用于：

- 功能开发
- Bug 修复
- 分支代码评审与吸收
- 测试补齐
- 文档同步

## 0. 指令优先级与冲突处理

当多个来源的规则同时存在时，默认按以下顺序执行：

1. system / developer 指令
2. 用户当前这一轮的明确任务要求
3. 仓库内专项任务文档 / 发布要求 / 验收文档
4. `AGENTS.md`
5. 本文档
6. 通用经验规则

如果两条规则冲突：

- 不允许自行“折中猜测”
- 必须优先遵守更高层规则
- 如冲突会明显影响实现范围或验收结论，应先明确说明再继续

## 1. 核心原则

### 1.1 先理解，再改代码

在开始实现前，必须先明确：

1. 用户真正要解决的问题是什么
2. 当前主线已经有什么能力
3. 这次改动会影响哪些已有行为
4. 成功标准是什么

不允许：

- 看到需求就直接大改
- 把猜测当成事实
- 用“看起来合理”的代码代替验证

### 1.2 最小可行改动

默认原则是：

- 只改完成当前任务必须改的代码
- 不主动清理无关代码
- 不顺手重构无关模块
- 不为了“未来可能扩展”提前加抽象

### 1.3 可验证优先于“感觉正确”

每个改动都要对应明确验证方式：

- 定向测试
- 全量测试
- 本地运行验证
- 必要时真实场景验证

不允许：

- 代码写完但没有验证
- 只靠逻辑推演就宣称“已修复”
- 把“理论上应该可行”当成“实际已通过”

## 2. 分支评审与代码吸收规则

### 2.1 不允许盲目合并旧分支

如果一个远端分支不是基于当前 `main`，默认不能直接 merge。

必须先看：

- `git log origin/main..origin/<branch>`
- `git diff --stat origin/main..origin/<branch>`

如果发现：

- 涉及大量无关文件
- 会回退主线已有能力
- 混入旧版本代码
- 文档或流程文件大面积倒退

则必须采用“安全吸收”策略，而不是直接合并。

### 2.2 安全吸收的默认方式

允许的方式只有两种：

1. `cherry-pick` 单个有价值提交，再在当前 `main` 上修正回归
2. 手动摘取有价值的改动，在当前 `main` 上重做

### 2.3 评审后再合入

分支代码不是“拉下来就合”，而是要先回答三个问题：

1. 这笔改动解决了什么问题
2. 它会不会破坏当前主线已有行为
3. 它有没有最小测试可以锁住

如果有明显回归风险，必须先修正再合入。

## 3. 代码修改规则

### 3.1 优先保持稳定接口

对外行为能兼容就兼容。优先新增，不轻易重命名或删除稳定结构。

### 3.2 具体规则优先于通用规则

当代码里同时存在通用匹配逻辑和更具体、更高置信度的规则时，必须让具体规则优先。

### 3.3 增强逻辑不能反向污染主行为

增强类改动只能提供更多信息、提高稳定性、改善提示质量，不能让已有主流程变得更脆弱。

### 3.4 高风险行为必须克制

点击、输入、表单提交、标签页切换、浏览器恢复、下载上传、扩展与 daemon 通讯这类高风险行为默认要更保守：

- 不重复触发动作
- 不模糊升级目标
- 不在不确定时做高风险 fallback

## 4. 测试规则

### 4.1 每次改动都要有最小回归测试

改动越具体，测试越具体；但至少要有一条测试能证明“这次问题以后不会再回来”。

### 4.2 先跑定向测试，再跑更大范围验证

推荐顺序：

1. 先跑和当前改动直接相关的测试
2. 再跑对应模块/包的验证
3. 必要时再跑仓库级门禁

### 4.3 区分代码级验证与真实场景验证

必须明确说明当前验证属于：

- 代码级验证
- 集成级验证
- 本机真实场景验证
- 真实用户路径验证

不能把“测试通过”表述成“真实现场已证明没问题”。

### 4.4 先区分“历史 CI 失败”与“当前主线失败”

查看 GitHub 提交列表、Actions 列表或 PR 检查状态时，必须先分清历史提交失败和当前 HEAD 失败。

## 5. 文档与协作规则

### 5.1 公开仓库优先看公开文档

在公开仓库里协作时，默认先看：

1. `AGENTS.md`
2. `README.md`
3. `docs/README.md`
4. `docs/PROJECT_STRUCTURE.md` / `docs/PROJECT_STRUCTURE_zh.md`
5. `docs/ARCHITECTURE.md` / `docs/ARCHITECTURE_zh.md`
6. `docs/TRANSPORT.md`
7. `docs/TOOLS.md` / `docs/TOOLS_zh.md`
8. `docs/TROUBLESHOOTING.md` / `docs/TROUBLESHOOTING_zh.md`

### 5.2 不把内部治理文档重新塞回公开树

公开 GitHub 仓库主要承载：

- 用户文档
- 开发者接入文档
- 贡献指南
- 架构与结构说明
- 发布、安全与合规说明

不要在公开树里重建内部 PM 系统、内部评审记录、夜测报告、验收资产或私有风险台账。

### 5.3 任务结束必须给清晰结论

每次任务结束时，应至少说明：

1. 改了什么
2. 验证了什么
3. 没验证什么
4. 还剩什么风险

## 6. 扩展与本地验收补充

当任务涉及 Chrome 扩展或真实浏览器行为时，默认本地验收链路是：

1. `pnpm -C app/chrome-extension build`
2. `pnpm run extension:reload`
3. 在真实浏览器里验证行为

如果没有 reload unpacked extension，就不要声称“浏览器侧已验证”。

## 7. 内部补充说明

如果你在维护者本地环境中工作，可能还会看到更细的内部流程文档和治理记录。

那些内部文档用于维护者协作，但不改变本公开规则文档的默认要求。
