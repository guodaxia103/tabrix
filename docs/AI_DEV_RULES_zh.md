# Tabrix AI 助手开发通用规则

本文档是 Tabrix 项目的**全局通用开发规则**。  
目标不是限制具体实现方式，而是让任何 AI 助手进入仓库后，都能以一致、稳定、低回归的方式工作。

这份规则适用于：

- 功能开发
- Bug 修复
- 分支代码评审与吸收
- 测试补齐
- 文档同步

## 1. 核心原则

### 1.1 先理解，再改代码

在开始实现前，必须先明确：

1. 用户真正要解决的问题是什么
2. 当前主线已经有什么能力
3. 这次改动会影响哪些已有行为
4. 成功标准是什么

不允许：

- 看到需求就直接大改
- 把猜测当成事实
- 用“看起来合理”的代码代替验证

### 1.2 最小可行改动

默认原则是：

- 只改完成当前任务必须改的代码
- 不主动清理无关代码
- 不顺手重构无关模块
- 不为了“未来可能扩展”提前加抽象

判断标准：

- 每一处改动都应该能直接对应当前任务
- 如果一段代码删掉后仍不影响这次目标，那它大概率不该出现在这次提交里

### 1.3 可验证优先于“感觉正确”

每个改动都要对应明确验证方式：

- 定向测试
- 全量测试
- 本地运行验证
- 必要时真实场景验证

不允许：

- 代码写完但没有验证
- 只靠逻辑推演就宣称“已修复”
- 把“理论上应该可行”当成“实际已通过”

## 2. 分支评审与代码吸收规则

### 2.1 不允许盲目合并旧分支

如果一个远端分支不是基于当前 `main`，默认不能直接 merge。

必须先看：

- `git log origin/main..origin/<branch>`
- `git diff --stat origin/main..origin/<branch>`

如果发现：

- 涉及大量无关文件
- 会回退主线已有能力
- 混入旧版本代码
- 文档或流程文件大面积倒退

则必须采用“安全吸收”策略，而不是直接合并。

### 2.2 安全吸收的默认方式

允许的方式只有两种：

1. `cherry-pick` 单个有价值提交，再在当前 `main` 上修正回归
2. 手动摘取有价值的改动，在当前 `main` 上重做

默认目标是：

- 保留分支里真正有价值的部分
- 不把旧分支的历史包袱一起带进来

### 2.3 评审后再合入

分支代码不是“拉下来就合”，而是要先回答三个问题：

1. 这笔改动解决了什么问题
2. 它会不会破坏当前主线已有行为
3. 它有没有最小测试可以锁住

如果有明显回归风险，必须先修正再合入。

## 3. 代码修改规则

### 3.1 优先保持稳定接口

对外行为能兼容就兼容。

如果要增强：

- 优先新增字段
- 尽量不重命名稳定字段
- 尽量不删除调用方已依赖的结构

只有在明确需要 breaking change 时，才允许调整既有语义。

### 3.2 具体规则优先于通用规则

当代码里同时存在：

- 通用匹配逻辑
- 更具体、更高置信度的规则

必须让**具体规则优先**。

否则常见结果是：

- 页面识别误判
- 目标元素识别错误
- 本来正确的具体场景被通用逻辑覆盖

### 3.3 增强逻辑不能反向污染主行为

增强类改动包括但不限于：

- 页面理解
- 元素分类
- 安全提示
- 页面 settle 元信息
- 诊断字段

这些增强只能：

- 提供更多信息
- 提高稳定性
- 改善提示质量

不能：

- 让已有主流程变得更脆弱
- 把非关键误判升级成关键路径失败
- 因新增判断而破坏已有正常路径

### 3.4 高风险行为必须克制

对以下类型的改动要格外保守：

- 点击/输入/表单提交
- 标签页切换
- 浏览器恢复与自动启动
- 原生对话框处理
- 下载/上传
- 扩展与 daemon 通讯

默认要求：

- 不重复触发动作
- 不模糊升级目标
- 不在不确定时做高风险 fallback

## 4. 测试规则

### 4.1 每次改动都要有最小回归测试

原则是：

- 改动越具体，测试越具体
- 不用为了一个小改动补一大堆无关测试
- 但必须有一条测试能证明“这次问题以后不会再回来”

### 4.2 先跑定向测试，再跑更大范围验证

推荐顺序：

1. 先跑和当前改动直接相关的测试
2. 再跑对应模块/包的验证
3. 必要时再跑仓库级门禁

这样可以更快定位问题，也能避免大范围测试掩盖根因。

### 4.3 测试失败时，不要硬解释成“测试问题”

先判断失败属于哪类：

1. 新代码真的有问题
2. 测试假设过期
3. 平台差异
4. 历史测试与新语义不一致

只有明确是测试假设过期时，才允许修测试。
否则先修代码。

### 4.4 区分代码级验证与真实场景验证

必须明确说明当前验证属于哪一层：

- 代码级验证
- 集成级验证
- 本机真实场景验证
- 真实用户路径验证

不能把“测试通过”表述成“真实现场已证明没问题”。

### 4.5 先区分“历史 CI 失败”与“当前主线失败”

查看 GitHub 提交列表、Actions 列表或 PR 检查状态时，必须先分清：

- 某个历史提交当时失败过
- 当前 `main` / 当前头提交仍然失败

GitHub 会保留每个 commit SHA 自己当时的检查结果。  
因此：

- 旧提交的红叉不会因为后续修复而自动变绿
- 看到“最近几笔提交有红叉”不等于“当前主线仍然坏着”

默认处理顺序：

1. 先确认当前头提交 SHA
2. 再确认当前头提交对应的最新 checks / workflow run
3. 必要时在本地重跑当前主线门禁
4. 只有确认当前头提交仍失败，才把它当成当前阻塞

不允许：

- 把历史失败记录误判为当前主线故障
- 为了“洗掉旧红叉”而制造没有实际价值的空提交
- 在没有核对当前头提交状态前就宣称“又坏了”

### 4.6 按改动类型执行最小测试矩阵

不要每次都机械地跑同一套命令，但也不能“只跑了一条小测试就提交”。

默认规则：

1. **纯文档改动**
   - 至少检查改动范围是否只包含文档
   - 如仓库有文档格式化钩子，确保提交前钩子通过

2. **单文件 / 单模块逻辑修复**
   - 先跑与当前改动直接对应的定向测试
   - 再跑对应包/模块的测试

3. **扩展代码改动**
   - `pnpm -C app/chrome-extension test`
   - `pnpm -C app/chrome-extension typecheck`
   - 如果改动影响真实浏览器行为，还必须重新 build 并在浏览器里 reload 后再做现场验证

4. **native-server / MCP 服务改动**
   - 至少跑对应 `jest` 定向测试
   - 再跑 `pnpm -C app/native-server test:ci`
   - 必要时补 `pnpm -C app/native-server build`

5. **跨 extension + native-server 的链路改动**
   - `pnpm run test:core`
   - 必要时再跑 `pnpm run typecheck`
   - 如果涉及发布/安装/运行一致性，还要做本机真实链路验证

6. **版本发布、安装流程、平台兼容、桥接恢复等高风险改动**
   - `pnpm run typecheck`
   - `pnpm run test:core`
   - 必要时 `pnpm run audit`
   - 必要时 `RELEASE_TAG=<tag> pnpm run release:check`

原则：

- 改动越靠近产品主链路，测试矩阵越不能省
- 如果声称“本机已经修好”，必须补对应的本机真实验证
- 如果只完成了代码级测试，就只能说代码级通过，不能说现场已通过

## 5. 提交与工作区规则

### 5.1 只提交本次相关文件

提交前必须检查：

- 本次任务相关文件
- 本地已有无关改动
- 临时文件
- 文档草稿

原则：

- 不把 unrelated 文件顺手带进来
- 不把临时目录、验收产物、草稿文件提交进仓库

### 5.2 测试通过后尽快提交

不要长期堆积本地未提交改动。

推荐节奏：

1. 完成一个小闭环
2. 测试通过
3. 立即提交

这样更容易：

- 回溯问题
- 缩小回归范围
- 降低合并风险

### 5.3 提交信息必须能说明意图

提交信息应清楚说明：

- 新功能
- 回归修复
- 测试对齐
- 文档同步

不推荐模糊提交信息。

### 5.4 运行时变更要确认“新代码真的在跑”

在 Tabrix 项目里，`源码已改`、`build 已完成`、`本机运行实例已切到新代码` 是三件不同的事。

必须明确区分：

1. **源码改了**
   - 只代表工作区文件已更新

2. **build 了**
   - 只代表新的 `dist` / `.output` 产物已生成

3. **运行实例切到新代码**
   - 需要服务重启、扩展 reload、或重新部署后才成立

不允许：

- 只改了源码就说“本机服务已经是最新”
- 只 build 了扩展就说“浏览器里已经跑到这版代码”
- 没确认运行实例一致性，就声称现场问题已复现或已修复

## 6. 文档规则

### 6.1 文档是能力的一部分，不是附属品

当改动影响以下内容时，应同步文档：

- 安装/启动行为
- 用户可见输出
- CLI 行为
- 错误码与诊断
- 平台差异
- 发布流程

### 6.2 文档更新也要讲边界

文档里必须区分：

- 已真实验证的能力
- 代码层支持但未做真实动态验收的能力
- 已知限制与待补项

不能为了好看把“理论支持”写成“完全稳定支持”。

## 7. 平台与环境规则

### 7.1 平台差异要显式处理

涉及 Windows / Ubuntu / macOS 的逻辑时，必须考虑：

- 路径差异
- 桌面会话差异
- 自启机制差异
- 浏览器安装方式差异

不能默认一个平台的行为在所有平台都成立。

### 7.2 Docker 不是所有平台问题的真实替代

允许用 Docker 验证：

- 安装
- CLI
- daemon
- 基础浏览器探测

但必须明确：

- Docker 不等于真实桌面环境
- 尤其不能替代真实 macOS 桌面行为验证

### 7.3 本地 MCP 服务的 build / 部署 / 重启 / 注册规则

涉及 `app/native-server`、桥接状态机、CLI、安装流程、浏览器恢复、daemon 的改动时，必须明确当前是在验证哪一层：

1. **工作区构建层**
   - 运行的是仓库里的 `dist`
   - 适合本地开发验证

2. **本机已安装服务层**
   - 运行的是全局安装或本机部署后的 CLI / 服务
   - 适合验证真实用户安装体验

默认规则：

- 改完 `native-server` 代码后，至少先 `build`
- 如果验证 daemon / status / doctor / 本机 HTTP MCP 服务，通常需要：
  - 重启 daemon
  - 或重新启动当前服务实例
- 只有在以下情况，才需要重新 `register`：
  - native host manifest 路径变了
  - 宿主程序入口变了
  - 系统级注册缺失或损坏

不允许：

- 每次改 native-server 都机械地重复 register
- 需要重启服务时却只重跑命令，不重启实例
- 没确认当前跑的是工作区版本还是全局安装版本，就下结论

推荐在需要时使用：

- `tabrix daemon status`
- `tabrix daemon start`
- `tabrix doctor --fix`
- `tabrix status --json`

来确认当前服务是否真切到新代码、桥接是否真的 ready

## 8. 浏览器自动化规则

### 8.1 优先安全路径

默认顺序：

1. 结构化读取
2. ref-based 操作
3. 辅助截图确认
4. 高风险 fallback（如计算机坐标、原始脚本）

不要一开始就走最激进的方式。

### 8.2 不允许“伪通过”

对以下场景尤其要严格：

- 原生 dialog
- 下载
- 上传
- 页面切换后读取
- 自动恢复

如果工具本身应该原生成功，就不能用注入绕过、页面改写、假事件模拟等方式把结果“做出来”然后当成通过。

### 8.3 扩展 build / 打包 / reload 规则

涉及 `app/chrome-extension` 的改动时，默认流程必须清楚：

1. **代码改动**
2. **重新 build**
3. **浏览器里 reload unpacked 扩展**
4. **再做真实浏览器验收**

必须明确：

- `.output/chrome-mv3` 是产物目录，不会因为源码变了就自动更新浏览器里的已加载扩展
- 浏览器里的 unpacked 扩展不会因为仓库重新 build 就自动切到新代码

不允许：

- 没 build 就说扩展已更新
- build 了但没 reload，就说浏览器现场已验证通过
- 把“代码级测试通过”误说成“浏览器扩展运行中已通过”

如果任务涉及发布产物，还要区分：

1. **本地开发 build**
   - 用于本机 reload 与开发验收

2. **发布打包**
   - 用于 GitHub Release / zip 产物 / 用户安装

这两层都要说明清楚，避免把开发 build 和发布包混为一谈

## 9. 与用户协作的规则

### 9.1 先说清真实状态

如果事情只完成到一半，要明确说：

- 哪一步已完成
- 哪一步还没验证
- 当前阻塞是什么

不要为了“看起来推进了很多”而模糊表达。

### 9.2 风险要提前说

如果一个操作存在明显风险，例如：

- 会把旧分支历史一起带进来
- 可能覆盖无关改动
- 可能引入大范围回退

必须先提醒，再继续。

### 9.3 建议要尽量可执行

给用户的建议应该是：

- 明确下一步
- 范围可控
- 不让用户承担不必要的排查成本

### 9.4 浏览器可见问题优先用 Tabrix 自己核查

对于以下类型的问题，默认优先使用 Tabrix 浏览器链路查看真实页面：

- GitHub Actions 失败页
- PR 页面与 review comments
- Issue 页面
- 登录后可见的网页状态
- 任何需要点击展开、切换标签、读取页面真实渲染结果的网页问题

这样做的目的有两个：

1. 按真实用户视角确认页面状态
2. 顺便把 Tabrix 当成真实验收场景持续使用

只有在以下情况，才优先改用 API / CLI：

- 需要批量结构化数据
- 需要稳定 JSON 输出
- 页面内容不足以定位问题
- 权限或交互限制导致网页路径效率明显更低

默认策略：

1. 先用 Tabrix 看网页
2. 再按需要补 API / CLI

不允许：

- 能用 Tabrix 看清的问题，完全跳过 Tabrix 直接用别的通道
- 用 API 结果替代网页真实状态，再把它当成网页已验证

## 10. 标准工作流

任何 AI 助手在 Tabrix 仓库内，默认按以下顺序工作：

1. 同步远端并确认当前分支状态
2. 理解问题与成功标准
3. 评估是否存在旧分支/旧主线风险
4. 只做最小必要改动
5. 补最小回归测试
6. 先跑定向测试，再跑更大范围验证
7. 只提交本次相关文件
8. 测试通过后立即提交
9. 必要时同步文档

## 11. 一句话纪律

> 不盲合、不猜测、不伪通过、不带无关文件；先验证，再提交。
