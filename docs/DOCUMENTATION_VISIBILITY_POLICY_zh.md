# Tabrix 文档公开与内部边界说明

`Tabrix` 面向未来商业化与开源协作并行发展。

因此仓库文档采用双层策略：

## 1. 适合放在 GitHub 的公开文档

公开文档用于服务：

- 用户接入
- 开发者集成
- 贡献者协作
- 发布与安全说明

典型包括：

- README / 安装 / 快速开始
- CLI / Tools / Transport / Troubleshooting
- Architecture / Project Structure
- Release process / Security / Changelog
- 第三方复用规范与合规说明

## 2. 不放在 GitHub 的内部文档

以下内容默认只保留在本地内部目录与飞书：

- 产品任务系统
- 执行看板
- AI 执行提示词
- 产品决策日志
- 版本装箱与任务分配
- 风险与依赖台账
- 证据索引与验收资产目录
- 夜测报告
- 内部评审、裁剪、路线图与优先级材料

原因：

- 避免泄露内部节奏、短板与路线判断
- 避免暴露 AI 协作与内部执行细节
- 让 GitHub 保持“面向外部协作者”的高信噪比

## 3. 公开文档的判断标准

适合公开：

- 对外可解释
- 对用户或贡献者有直接价值
- 不暴露内部优先级、验收细节和经营判断

不适合公开：

- 内部路线图
- 任务拆解与状态管理
- 内部验收、夜测、证据留痕
- AI 提示词与内部协作规则细节

## 4. 维护原则

- GitHub `docs/` 只保留公开文档
- 内部 PM / 策略 / 验收文档留在本地内部目录与飞书
- 如果某份内部文档未来需要公开，应先做“公开版重写”，而不是直接原样发布
